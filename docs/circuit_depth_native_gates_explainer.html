Circuit Depth, Native Gates, and the IonQ vs Heavy-Hex Comparison

This note explains what circuit depth means in a hardware-aware setting, why depth depends on the target native gate set, and how the comparison between an IonQ-style all-to-all target and a heavy-hex target is constructed. The goal is not a code walkthrough. The goal is to make the physics and the compiler choices legible.

The central point is simple: compiled depth is not a hardware-independent quantity. A fixed logical time-evolution circuit can acquire different total depth on different targets even when the two-qubit interaction count is identical. That happens because the compiler is decomposing the same logical unitary into different native gate alphabets.

1. What “Depth” Means Here

Depth is the number of sequential gate layers after the circuit has been expressed in a chosen basis and, if necessary, routed onto a chosen connectivity graph. In practice, depth depends on three distinct ingredients:

Logical structure

The physics chooses the Hamiltonian terms and the Trotter order. This fixes the abstract quantum operations that must be implemented.

Native gate set

The same logical operation may decompose into different sequences of one- and two-qubit gates depending on the hardware basis.

Connectivity and routing

If the hardware graph does not directly support a required interaction, the compiler may add SWAPs or alternative rewrites, increasing depth and two-qubit cost.

For that reason, the phrase “IonQ has higher depth” must always be unpacked into:

  • higher logical depth, or
  • higher native compiled depth, or
  • higher two-qubit depth, or
  • higher routing overhead.

Those are not interchangeable. In particular, a backend can have the same two-qubit gate count and zero SWAP overhead, yet still show larger total depth because it uses more one-qubit basis-change layers around the entangling operations.

2. The Physics Being Simulated

The comparison is built around digital Hamiltonian simulation using a first-order Lie-Trotter product formula. The code supports transverse-field Ising and Heisenberg-type models, with interaction graphs such as nearest-neighbor chain, all-to-all, and power-law couplings.

U(t) = e-iHt ≈ ( Πk e-iHk Δt )r, where Δt = t / r.

The important implication is that the physics first produces a list of Pauli-string terms. Only after that does hardware enter the picture. The comparison is therefore:

  1. build one logical Trotter circuit for the same Hamiltonian,
  2. transpile it to an IonQ-compatible native basis,
  3. transpile the same logical circuit to a heavy-hex target,
  4. measure depth and gate statistics on the resulting compiled circuits.

3. How a Pauli Term Becomes Gates

Each term exp(-i c Δt P) is implemented by a standard Pauli-rotation pattern:

  • basis change to map X or Y support into a Z-frame,
  • a CNOT staircase that accumulates parity,
  • a single Rz rotation on the terminal active qubit,
  • uncomputation of the staircase and basis change.
for coeff, pauli_str in hamiltonian_terms:
    apply basis changes
    apply a CNOT chain across active qubits
    apply Rz(angle)
    undo the CNOT chain
    undo basis changes

From a physics perspective, this is still the same Trotterized evolution. From a hardware perspective, it is only an intermediate representation. The transpiler then rewrites that circuit into the native basis of the chosen target.

4. Why Native Gates Change Depth

In this comparison, the IonQ-oriented target is compiled to a basis equivalent to {rx, ry, rz, rxx}, reflecting an all-to-all trapped-ion style entangling model. The heavy-hex target is compiled to a basis equivalent to {rz, sx, x, cx, swap}, reflecting a superconducting architecture with constrained connectivity.

Target Native-style basis used in the comparison Compiler implication
IonQ-style all-to-all rx, ry, rz, rxx Routing pressure is minimized, but single-qubit basis-change layers may expand total depth.
Heavy-hex rz, sx, x, cx, swap Basis conversion can be compact, but nonlocal interactions may require explicit routing.

This is why total depth alone is not a pure connectivity metric. A higher IonQ depth does not automatically mean the circuit is harder for IonQ in the sense that matters physically. It may simply mean that the native representation carries more sequential one-qubit structure.

If two targets have the same two-qubit gate count and the same two-qubit depth, but one has larger total depth, the most natural interpretation is basis-decomposition overhead, not routing overhead.

5. How the Comparison Is Actually Constructed

Conceptually, the comparison pipeline is: 

logical_circuit = build_problem_circuit(...)
ionq_circuit = transpile_for_ionq(logical_circuit, optimization_level=2)
heavy_hex_circuit = transpile_for_heavy_hex(logical_circuit, optimization_level=2)

ionq_summary = connectivity_circuit_summary(ionq_circuit)
heavy_hex_summary = connectivity_circuit_summary(heavy_hex_circuit)

The same logical circuit is used on both sides. No physics is changed between the two branches. Only the target basis and connectivity assumptions are changed.

Metrics extracted from each compiled circuit

  • Depth: total compiled circuit depth.
  • Two-qubit gates: count of gates such as cx, rxx, swap, and related entangling operations.
  • SWAP count: explicit routing overhead exposed as inserted SWAP gates.
  • Two-qubit depth: number of layers that contain any two-qubit operation.

Two-qubit depth is especially useful because it isolates the sequential entangling structure from the surrounding one-qubit dressing.

6. Key Qiskit Functions and the Parameters That Matter

The following are the important library-level operations. This is the shortest accurate list for understanding the comparison.

Function Library Important parameters Role in the comparison
QuantumCircuit(...) Qiskit n_qubits, classical bits Creates the logical Trotter circuit and the measured execution circuit.
QuantumCircuit.rx, ry, rz, cx, h, x, measure Qiskit rotation angle, qubit indices Builds the explicit Pauli-term decomposition before backend-specific transpilation.
transpile(...) Qiskit basis_gates, coupling_map, optimization_level, layout_method, routing_method Rewrites the same logical circuit into the target basis and topology.
CouplingMap.from_heavy_hex(distance) Qiskit transpiler distance Builds the heavy-hex connectivity graph used as the routing constraint.
QuantumCircuit.depth() Qiskit optional filter_function Computes either total depth or, with a filter, depth restricted to two-qubit layers.
QuantumCircuit.count_ops() Qiskit none Returns the gate histogram used to derive two-qubit counts and SWAP counts.
circuit_to_dag(...) Qiskit converters circuit Fallback path for layer-by-layer two-qubit depth analysis if direct filtered depth is unavailable.

The two transpilation calls that define the hardware comparison

transpile(
    circuit,
    basis_gates=["rx", "ry", "rz", "rxx", "id", "measure"],
    optimization_level=2
)
transpile(
    circuit,
    basis_gates=["rz", "sx", "x", "cx", "swap", "measure"],
    coupling_map=CouplingMap.from_heavy_hex(distance),
    optimization_level=2,
    layout_method="sabre",
    routing_method="sabre"
)

The IonQ-style branch is unconstrained in connectivity. The heavy-hex branch is explicitly constrained and routed.

7. How to Read the Output Correctly

The most robust interpretation is metric-by-metric rather than headline-by-headline:

If you observe Most likely interpretation
higher total depth, same two-qubit count, same SWAP count native basis decomposition is different; not evidence of worse connectivity
higher SWAP count routing overhead is being paid explicitly
higher two-qubit depth the entangling schedule itself has become more sequential
higher two-qubit count and higher SWAP count together routing is not just scheduling the same entanglers; it is adding entangling work

For a nearest-neighbor chain, it is entirely plausible to see:

  • zero SWAPs on both targets,
  • the same two-qubit gate count on both targets,
  • yet a larger IonQ total depth.

In that regime, the difference is telling you more about how the circuit was expressed in {rx, ry, rz, rxx} versus {rz, sx, x, cx} than about the physical interaction graph.

8. The Right Mental Model

The clean mental model is:

  • the Hamiltonian and Trotter step count define the quantum workload,
  • the native gate set defines how that workload is spelled out physically,
  • the coupling map defines whether extra routing must be inserted,
  • the reported depth is the combined result of all three.
A hardware-aware depth comparison is meaningful only when paired with gate counts and a statement of the native basis. Otherwise, one is comparing two compiler dialects rather than two physical implementations of the same logical circuit.

9. Bottom Line

This comparison is not asking which platform has the universally smaller depth. It is asking how the same Trotterized quantum evolution is rewritten under two different hardware models. The IonQ-style compilation removes connectivity constraints, but the compiled circuit still reflects the native gate alphabet chosen for trapped-ion execution. Therefore, a larger IonQ total depth can be consistent with no routing disadvantage at all.

The most decision-relevant quantities are usually: two-qubit gate count, two-qubit depth, and SWAP count. Total depth remains useful, but only when interpreted through the lens of the target native basis.
