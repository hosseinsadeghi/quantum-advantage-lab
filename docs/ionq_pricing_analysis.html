IonQ QPU pricing & qubit-scaling analysis

Quantum Advantage Lab — access-path and budget assessment

Captured 2026-04-18. Pricing data is fetched from vendor pages and can drift — verify before committing spend.

TL;DR. Aria-1 and Aria-2 QPUs are retired; their calibration snapshots remain as selectable emulator profiles on ionq_simulator. The only production IonQ QPU is Forte-1 (36 qubits). Pricing splits into two models: AWS Braket uses flat per-shot billing; Azure Quantum and IonQ direct cloud both use a gate-shot (AQT) model with per-program minimums. For the depth/width regime this project targets (N=20–30, Trotter depth 8–16), Braket is consistently cheaper per run; the AQT model is favorable only for small, shot-heavy circuits.

Azure Quantum and IonQ direct — gate-shot (AQT) pricing

Both paths charge Azure Quantum Tokens (AQT) for usage:

AQT = m + 0.000220·(N_1q·C) + 0.000975·(N_2q·C)

m is the per-program minimum ($168.195 with error mitigation on / $25.7899 off for Forte-1; Aria-1 uses $97.50 / $12.4166). N_1q and N_2q are the 1- and 2-qubit gate counts; C is the shot count. Multi-controlled 2q gates are billed as 6·(N−2) 2q gates; 1q gates are internally priced as 0.225 of a 2q gate.

4. Where classical starts to struggle

Exact statevector matrix-exponentiation (our current classical/matrix_exp.py baseline):

Runtime grows worse than memory (naïve expm is O(23N)). N≈25–30 is the visible "classical is losing" regime for dense Hamiltonian evolution on commodity hardware. Short-shallow circuits can still be simulated at higher N with MPS, but the long-time quench and deep-Trotter runs we want to showcase are precisely where MPS bond dimension blows up — favorable for the demo.

6. Recommendations

  • Default user-facing runs to the emulator. The free ionq_simulator with noise_model in {ideal, forte-1, aria-1, aria-2, …} covers all correctness, benchmarking, and UX testing at zero cost.
  • Target N=20–30 in UI defaults, not N=3–4. Reserve low-N as a "quick demo" preset only — it's trivially classical and does not support the quantum-advantage narrative.
  • Bias toward AWS Braket for hero QPU runs if access is granted there: flat-shot pricing is dramatically cheaper for the deep, wide circuits we'll showcase.
  • Gate the QPU path behind an explicit UI action ("Run on hardware") with a pre-flight cost estimate. Keep use_qpu=False as the default for all API calls.
  • Add a cost estimator to scripts/ionq_run_hamiltonian.py that reports (N_1q, N_2q, shots) and the computed AQT / Braket cost before any submission.
  • Keep the provider plumbing path-agnostic: backend name is a string, noise model is a string, use_qpu is a boolean. This makes switching between Braket, Azure Quantum, and IonQ direct a one-line change once the credit path is confirmed.
